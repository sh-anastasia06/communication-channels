### Политика разработки проекта (Spring Boot + JDBC) ###

1. Архитектура слоев
В проекте строго соблюдается принцип Layered Architecture с использованием интерфейсов:
Controller ? Service Interface ? Service Implementation ? Repository Interface ? Repository Implementation.
Вся логика описывается в Service Implementation.

2. Модели данных (DTO / Records)
Для передачи данных используются Java Records.
Иммутабельность: Данные внутри DTO не могут быть изменены.
Перегруженные конструкторы: Обязательно использование специализированных конструкторов для разных сценариев (SELECT, INSERT, UPDATE).
Маппинг: В конструкторах для частичного заполнения неиспользуемые поля инициализируются как null через this(...).

3. Слой данных (Repository Implementation)

Реализация строится на NamedParameterJdbcTemplate.
Требования к коду:
- Constructor Injection: Все зависимости (Keycloak, JdbcTemplate, ObjectMapper, NamedParameterJdbcTemplate) внедряются через 
конструктор и помечаются private final.
- Динамический SQL:
Использование StringBuilder для построения запроса.
Базовое условие WHERE 1=1 для удобной конкатенации фильтров.
Использование оператора ILIKE :param для регистронезависимого поиска.
- Безопасность: Запрещено использование конкатенации строк для значений параметров (только именованные параметры :id).

4. Централизованная обработка ошибок (Global Exception Handling)

В проекте реализован механизм перехвата исключений на уровне контроллеров, что гарантирует получение 
клиентом стандартизированного ответа ErrorResponseDTO.



### Политика именования API Endpoints ###

В проекте используется автоматическая настройка прав доступа на основе паттернов URL. 
Чтобы ваш эндпоинт корректно прошел проверку безопасности, он **должен** начинаться с одного из зарезервированных префиксов.

| Доступ           | Префикс         | Описание                       | Пример URL          |
|:-----------------|:----------------|:-------------------------------|:--------------------|
| **Admin Only**   | `/api/new*`     | Создание новой записи          | `/api/newconsumer`  |
| **Admin Only**   | `/api/update*`  | Обновление существующих данных | `/api/updatenode`   |
| **Admin Only**   | `/api/del*`     | Удаление записей               | `/api/dellocation`  |
| **User & Admin** | `/api/all*`     | Получение списков (чтение)     | `/api/allnodes`     |
| **User & Admin** | `/api/filter/*` | Получение выпадающих списков   | `/api/filter/nodes` |

1. **Регистр:** Только нижний регистр (`lowercase`).
2. **Расположение:** Префикс должен идти сразу после `/api/`.
    * ? Правильно: `/api/updateuser`
    * ? Неправильно: `/api/user/update` (не попадет в шаблон `/api/update*`)
3. **Контроллеры:** При использовании `@RequestMapping("/api")` на уровне класса, методы должны начинаться с префикса:
   @PostMapping("/newlocation") // Автоматически станет /api/newlocation



### Концепция логирования в проекте ###

В проекте реализовано централизованное логирование бизнес-методов с помощью Spring AOP (Aspect-Oriented Programming).
1. Аннотация @Loggable
   Для того чтобы метод начал логироваться, его достаточно пометить аннотацией @Loggable.
   Где применять: Рекомендуется помечать методы контроллеров.
   Что логируется: Автоматически фиксируется вход в метод, переданные параметры, результат выполнения и время работы.
2. Аспект LoggingAspect
   За обработку отвечает LoggingAspect. Он работает по следующему алгоритму:
   - START: При вызове метода в лог выводится его имя и все входящие аргументы.
   - SUCCESS: При успешном завершении выводится результат и время выполнения в секундах.
   - ERROR: Если метод выбросил исключение, в лог попадает текст ошибки (message) и затраченное до падения время.
